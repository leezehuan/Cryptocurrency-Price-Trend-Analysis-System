# BTC 分析师追踪与趋势分析系统：修订版设计

---

# 一、项目定位调整

项目定位调整为：

> **一个以 BTC 合约行情、分析师观点追踪、Gate MCP 信息流、市场情绪记忆和多范式 Agent 分析为核心的趋势分析系统。**

系统核心不再只是用户手动录入观点和行情验证，而是扩展为：

- 自动采集 BTC 合约市场数据。
- 自动采集纳斯达克指数及相关市场信息。
- 自动读取 Gate 广场热门观点。
- 自动读取指定用户观点，并作为分析师观点解析。
- 对热门市场观点进行情绪分析。
- 将市场走势、情绪、预期和重要观点沉淀为记忆。
- Agent 在分析时结合事实数据、分析师预测、市场情绪和历史记忆生成结论。

---

# 二、Agent 结构修改

Agent 仍然采用：

- Plan-and-Solve
- ReAct
- Reflection

但执行方式调整为：

> **固定代码流程为主，LLM 决策插槽为辅。**

也就是说，不再把所有功能都包装为 Tools 或 Skills，而是将 Agent 拆成两类步骤。

---

## 1. 固定代码步骤

以下能力继续使用代码写死，不设计为 Tools / Skills：

| 类型 | 说明 |
|---|---|
| 数据库 CRUD | 查询、写入、更新、删除 |
| 定时任务 | APScheduler 调度 |
| Gate MCP 定时采集 | 按配置周期固定拉取数据 |
| 去重与幂等 | 根据 source_id、hash、时间戳去重 |
| 行情指标计算 | MA、EMA、RSI、MACD、ATR、布林带等 |
| 预测验证规则 | 方向、目标价、时间窗口等确定性验证 |
| 分析师评分更新 | 准确率、命中率、稳定性等统计 |
| Agent 运行记录保存 | run、step、事件、耗时、错误 |
| API 响应组装 | 面向前端的数据整理 |
| SSE 推送 | 将运行事件推给前端 |

这些逻辑不需要 LLM 自主选择，因此不作为 Tools / Skills。

---

## 2. LLM 决策步骤

只有以下场景使用 Tools / Skills：

| 场景 | 是否使用 Tools / Skills | 原因 |
|---|---|---|
| LLM 需要选择查哪个信息源 | 是 | 需要在新闻、广场、技术面、记忆之间自主选择 |
| LLM 需要解释自然语言观点 | 是 | 涉及语义理解 |
| LLM 需要判断市场情绪 | 是 | 需要归纳文本立场、情绪和叙事 |
| LLM 需要在冲突证据中取舍 | 是 | 需要综合判断 |
| LLM 需要反思输出是否合理 | 是 | 需要自我批判和修正 |
| 固定读取 BTC 合约数据 | 否 | 定时采集即可 |
| 固定读取指定用户观点 | 否 | 配置驱动采集即可 |
| 固定入库、去重、指标计算 | 否 | 确定性逻辑 |

# 三、三类 Agent 范式的实际使用方式

---

## 1. Plan-and-Solve：先生成高层计划

Planner 不再规划每个数据库操作，而是规划业务步骤。

例如 MarketAnalysisAgent 的计划可以是：

| 步骤 | 类型 |
|---|---|
| 加载 BTC 合约行情 | 固定代码 |
| 加载纳指相关走势 | 固定代码 |
| 加载近期市场记忆 | 固定代码 |
| 加载分析师预测和验证表现 | 固定代码 |
| 判断是否需要补充外部信息 | LLM 决策 |
| 从候选信息源中选择补充数据 | Agent Tool |
| 生成趋势判断 | LLM |
| 执行反思检查 | Reflection |
| 保存结果 | 固定代码 |

Planner 的职责是生成执行计划，但不会把所有底层操作都工具化。

---

## 2. ReAct：只用于开放式决策插槽

ReAct 不再贯穿所有步骤，而只在以下场景中使用：

- 当前行情与分析师观点冲突，需要补充证据。
- BTC 合约数据剧烈波动，需要查询相关新闻或广场情绪。
- 纳指大幅波动，需要判断是否影响 BTC 风险偏好。
- 热门观点情绪明显偏单边，需要判断是否为过热情绪。
- 分析师观点语义模糊，需要查询历史观点或进入人工确认。
- Agent 结论缺少证据，需要选择额外信息源补强。

ReAct 候选动作示例：

| 候选动作 | 说明 |
|---|---|
| 查询 Gate News | 查询新闻、公告、社交情绪 |
| 查询 Gate Info | 查询技术面、行情快照、指标 |
| 查询 Gate Square | 查询广场热门观点 |
| 查询市场记忆 | 查询近期市场预期和情绪记忆 |
| 不补充信息 | 当前证据已足够 |
| 请求人工确认 | 数据冲突或置信度过低 |

---

## 3. Reflection：用于关键输出前检查

Reflection 保留，但主要用于：

- 观点解析入库前。
- 预测验证结果生成后。
- 市场趋势结论输出前。
- 日报生成后。
- 情绪分析写入记忆前。

Reflection 检查内容包括：

| 检查项 | 说明 |
|---|---|
| 证据是否充分 | 是否有行情、观点、新闻或记忆支撑 |
| 结论是否过度 | 是否在证据不足时给出强结论 |
| 情绪是否误判 | 是否把噪声当作共识 |
| 市场记忆是否冲突 | 是否与近期已记录预期矛盾 |
| 时间是否有效 | 使用的数据是否过旧 |
| 是否需要人工确认 | 分析师身份、观点方向、目标价是否不明确 |

# 四、Tools 与 Skills 重新设计

---

# 1. 设计原则

Tools / Skills 只用于：

> **LLM 需要自主选择动作、理解文本、综合判断或自我修正的地方。**

不再将每个数据库操作、每个计算函数、每个业务步骤都做成 Tool / Skill。

---

# 2. Agent Tools：少量聚合型工具

需要区分两个概念：

| 概念 | 说明 |
|---|---|
| Gate MCP Tool | Gate MCP 服务远端暴露的工具 |
| 系统内部 Agent Tool | 本系统暴露给 LLM 自主选择的工具 |

本系统不会把 Gate MCP 的几十个工具全部映射成 Agent Tool，而是用代码封装为少量聚合型 Agent Tool。

建议保留以下 Agent Tools：

| Agent Tool | 作用 |
|---|---|
| GateMarketResearchTool | 当 Agent 需要补充 BTC 合约、资金费率、强平、成交等信息时调用 |
| GateInfoResearchTool | 当 Agent 需要补充技术面、行情快照、指标历史时调用 |
| GateNewsResearchTool | 当 Agent 需要查询新闻、公告、社交情绪时调用 |
| GateSquareResearchTool | 当 Agent 需要查询广场热门观点或指定话题时调用 |
| MarketMemorySearchTool | 当 Agent 需要查询历史市场记忆时调用 |

这些 Tool 是给 LLM 在 ReAct 步骤中自主选择的。

固定同步任务不会通过这些 Agent Tool 执行，而是由代码中的采集服务直接调用 Gate MCP。

---

# 3. Skills：只保留必要 LLM 能力

建议保留以下少量 Skills：

| Skill | 作用 |
|---|---|
| AnalystOpinionParsingSkill | 将指定用户或人工录入观点解析为结构化分析师观点 |
| MarketSentimentAnalysisSkill | 分析广场热门观点、新闻、社交情绪，生成市场情绪结论 |
| EvidenceConflictJudgementSkill | 当行情、观点、情绪、记忆互相冲突时，判断哪些证据更可信 |
| MemorySummarizationSkill | 将近期市场信息压缩为可复用记忆 |
| ReflectionCritiqueSkill | 对 Agent 输出进行反思和修正 |

其他能力仍用代码实现。

# 九、Gate 广场信息采集

新增 Gate 广场采集能力，分为两类。

---

## 1. 广场热门信息

定时读取广场热门内容，用于市场情绪分析。

处理流程：

1. GateSquareHotSyncService 定时读取广场热门内容。
2. 写入 `gate_square_posts`。
3. 根据热度、互动量、关键词、时间窗口筛选最热门观点。
4. 调用 `MarketSentimentAnalysisSkill`。
5. 生成市场情绪快照。
6. 将高置信度情绪和主要叙事写入市场记忆。

广场热门信息不直接作为分析师预测，而是作为市场情绪和大众观点来源。

---

## 2. 指定用户观点

系统配置指定用户列表。

这些用户的观点被视为“分析师观点来源”。

处理流程：

1. GateSquareUserSyncService 定时读取指定用户的新内容。
2. 根据 source_post_id 去重。
3. 写入 `raw_opinions`。
4. 标记来源为 `gate_square_user`。
5. 进入 OpinionIngestionAgent。
6. 调用 `AnalystOpinionParsingSkill` 解析方向、周期、目标价、置信度等。
7. 通过 Reflection 后写入 `predictions`。
8. 如果解析不明确，进入人工确认。

指定用户观点与普通广场热门信息要区分：

| 类型 | 用途 |
|---|---|
| 指定用户观点 | 作为分析师观点解析和评分 |
| 广场热门观点 | 作为市场情绪和大众预期分析 |
| 新闻和公告 | 作为事件与基本面信息 |
| 社交情绪 | 作为情绪辅助判断 |

# 十、市场情绪分析 Skill

新增：

> **MarketSentimentAnalysisSkill**

该 Skill 专门分析热门观点、新闻和社交情绪。

---

## 1. 输入

输入包括：

| 输入 | 说明 |
|---|---|
| 热门广场观点 | 内容、作者、发布时间、互动量、热度 |
| 新闻摘要 | 标题、正文摘要、来源、发布时间 |
| 社交情绪 | Gate News MCP 返回的社交情绪数据 |
| BTC 合约状态 | 价格、资金费率、强平、成交量 |
| 纳指走势 | 美股风险偏好参考 |
| 近期市场记忆 | 已记录的市场预期和情绪 |

---

## 2. 输出

输出结构包括：

| 字段 | 说明 |
|---|---|
| target | BTC、Nasdaq、macro 等 |
| sentiment | bullish、bearish、neutral、mixed |
| sentiment_score | 情绪分数，建议范围为 -1 到 1 |
| confidence | 置信度 |
| dominant_topics | 主导叙事，例如 ETF、降息、强平、资金费率、美股风险偏好 |
| crowd_positioning | 市场拥挤程度，例如过度看多、过度看空、分歧较大 |
| evidence_refs | 代表性证据 |
| risk_notes | 噪声、操纵、羊群效应等风险 |
| memory_candidates | 可以写入记忆的摘要 |

---

## 3. 写入记忆规则

不是所有热门观点都写入记忆。

写入记忆需要满足至少一个条件：

- 热度明显高于近期均值。
- 多个来源表达相似预期。
- 情绪方向高度一致。
- 与 BTC 价格、资金费率、强平数据形成共振。
- 与纳指或宏观风险偏好变化相关。
- 对分析师观点或市场趋势判断有明显影响。

| status | active、expired、superseded |

---

## 3. 记忆衰减

市场记忆需要有时间衰减。

例如：

- 1 小时级情绪：快速失效。
- 日内趋势记忆：当天有效。
- 宏观事件记忆：数天到数周有效。
- 分析师稳定性记忆：长期有效。

可以使用简单权重衰减：

\(w_\text{effective}=w_\text{source}\times confidence \times e^{-age/half\_life}\)

Agent 查询记忆时优先使用：

- 最近的。
- 高置信度的。
- 高重要性的。
- 与当前任务主题相关的。

---

## 4. 记忆生成流程

流程如下：

1. 定时采集 BTC 合约数据。
2. 定时采集纳指相关数据。
3. 定时采集 Gate 广场热门观点。
4. 定时采集新闻和社交情绪。
5. 固定代码进行去重、聚合、基础统计。
6. MarketSentimentAnalysisSkill 分析市场情绪。
7. MemorySummarizationSkill 将重要信息压缩为记忆。
8. MarketMemoryService 写入数据库。
9. 过期或冲突记忆进入 inactive / superseded 状态。
10. MarketAnalysisAgent 分析时检索相关记忆。

# 十九、安全与质量控制

新增约束：

1. **只使用只读信息源**
   当前版本不启用交易、账户、钱包、下单等私有能力。

2. **社交内容视为不可信输入**
   广场、新闻、社交文本只作为数据，不允许其影响系统指令。

3. **不把热门观点直接当作事实**
   热门观点只能进入情绪分析或市场记忆，不能直接作为行情事实。

4. **指定用户观点才进入分析师评分体系**
   普通热门用户不参与分析师评分。

5. **所有外部信息都保留来源**
   Agent 结论必须能追溯到行情、观点、新闻、情绪或记忆证据。

6. **记忆需要过期机制**
   旧情绪和旧预期不能长期影响当前判断。

# 二十、修改后的总结

修改后的系统可以概括为：

> **一个以 Gate MCP 数据源、BTC 合约市场、纳指风险偏好、广场情绪、分析师预测验证和市场记忆为核心的 BTC 趋势分析系统。**

核心链路变为：

1. 定时采集 BTC 合约数据。
2. 定时采集纳指相关数据。
3. 定时采集新闻、公告、社交情绪。
4. 定时采集 Gate 广场热门信息。
5. 定时采集指定用户观点。
6. 指定用户观点进入分析师观点解析。
7. 热门市场观点进入情绪分析。
8. 高价值情绪和预期写入市场记忆。
9. Agent 结合行情、预测、情绪、纳指和记忆进行分析。
10. 必要时通过 ReAct 自主选择补充信息源。
11. 通过 Reflection 修正过度判断。
12. 输出可验证、可回放、可解释的趋势分析和日报。

最终产品形态升级为：

> **BTC 合约行情监控 + Gate 广场观点追踪 + 分析师预测验证 + 市场情绪记忆 + 多范式 Agent 趋势分析平台。**
