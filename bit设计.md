下面是**项目设计方案**，不包含具体编程实现。整体目标是：基于 LangGraph 做一个可监控 BTC 价格、收集信息、整理分析师观点、验证预测准确率、并进行虚拟合约交易回测/模拟的前后端分离系统。

---

# 一、总体架构

推荐架构：

前端 Web 应用  
→ 后端 API 服务  
→ LangGraph Agent 工作流  
→ 本地数据库  
→ 币安/新闻/用户输入/其他数据源

核心原则：

1. **前后端分离**
   - 前端只负责展示、交互、录入分析师观点。
   - 后端负责数据抓取、数据库、LangGraph、预测解析、验证、评分、虚拟交易。

2. **本地数据库优先**
   - BTC 历史行情、技术指标、分析师预测、验证结果、虚拟交易记录都存在本地。
   - 外部 API 只作为数据来源，不作为长期依赖。

3. **LLM 不直接决定最终结果**
   - LLM 用来总结、解析、归纳。
   - 价格验证、胜率计算、交易盈亏计算必须使用确定性规则。

4. **Harness Engineering 思想**
   - 给 Agent 加上“工程外壳”：可测试、可回放、可评估、可追踪、可审计。
   - 每一次输入、模型输出、解析结果、评分结果都要能追溯。

---

# 二、Harness Engineering 设计思想

这里的 Harness Engineering 可以理解为：

> 不让 Agent 自由发挥，而是把 Agent 放进一套可控、可测试、可评估的流程里。

## 1. 输入 Harness

所有用户输入的分析师观点，都先进入标准化流程：

- 原始内容保存
- 识别分析师
- 识别发布时间
- 识别预测方向
- 识别预测目标价
- 识别时间周期
- 识别条件和失效条件
- 生成结构化预测记录

例如用户输入：

> 某分析师说：BTC 短期可能会涨到 80000，中期可能回调到 75000。

系统应拆成两条预测：

| 周期 | 方向 | 目标价 | 验证时间 |
|---|---|---|---|
| 短期 | 看涨 | 80000 | 默认 7 天后，或用户指定 |
| 中期 | 看跌/回调 | 75000 | 默认 30 天后，或用户指定 |

---

## 2. 输出 Harness

LLM 的输出必须符合固定结构，例如：

- 分析师是谁
- 原文摘要
- 预测方向
- 目标价
- 时间周期
- 置信度
- 是否明确
- 是否需要人工确认

如果模型输出不完整，系统不能直接入库，需要进入人工确认流程。

---

## 3. 验证 Harness

价格预测不能靠模型主观判断，需要用数据库里的 BTC 实际行情验证。

验证内容包括：

1. 方向是否正确  
2. 目标价格是否接近或触达  
3. 时间是否符合  
4. 是否中途修改观点  
5. 是否存在模糊表达导致无法验证

---

## 4. 回放 Harness

系统需要支持：

- 回放过去某个分析师的所有预测
- 回放某一天 Agent 的判断
- 回放某个预测是如何被解析、评分、交易模拟的

这对后期调试很重要。

---

## 5. 评估 Harness

不仅看“胜率”，还要看：

- 方向准确率
- 目标价接近率
- 预测稳定性
- 修改观点频率
- 虚拟交易收益率
- 最大回撤
- 盈亏比

因为一个分析师可能方向经常对，但目标价很夸张；也可能预测胜率不高，但盈亏比很好。

## 5. 预测验证模块

系统需要在指定时间自动验证预测。

### 方向验证

设：

- $P_0$ = 预测发布时价格
- $P_1$ = 验证时价格

收益变化：

$$
r = \frac{P_1 - P_0}{P_0}
$$

如果：

- $r > 设定阈值$，判断为上涨
- $r < -设定阈值$，判断为下跌
- 否则判断为震荡

阈值建议使用：

- 固定阈值：例如 1%
- 或 ATR 波动阈值

---

### 目标价验证

例如分析师说：

> 短期涨到 80000

验证时不能只看最后一刻价格，也要看验证区间内是否触达。

如果是看涨目标价：

- 验证区间最高价 >= 目标价，则目标达成

如果是看跌目标价：

- 验证区间最低价 <= 目标价，则目标达成

目标接近率：

$$
s_{target} = \max\left(0, 1 - \frac{|P_{actual} - P_{target}|}{P_{target} \times \tau}\right)
$$

其中：

- $P_{actual}$ 是验证区间内最接近目标价的价格
- $P_{target}$ 是目标价
- $\tau$ 是容忍区间，例如 5%

---

### 综合评分

建议不要只用“成功/失败”，而是使用综合分：

$$
Score = 0.45 \times 方向分 + 0.40 \times 目标价分 + 0.15 \times 时间分 - 修改观点惩罚
$$

然后把预测分为：

| 分数 | 结果 |
|---|---|
| 0.8 - 1.0 | 高质量成功 |
| 0.6 - 0.8 | 基本成功 |
| 0.4 - 0.6 | 部分正确 |
| 0 - 0.4 | 失败 |

## 6. 分析师胜率模块

建议每个分析师维护多套指标：

| 指标 | 说明 |
|---|---|
| 硬胜率 | 成功次数 / 总预测次数 |
| 加权胜率 | 根据预测分数计算 |
| 方向准确率 | 只看方向是否对 |
| 目标价准确率 | 看目标价接近程度 |
| 修改观点率 | 中途变卦次数 |
| 短期胜率 | 短期预测表现 |
| 中期胜率 | 中期预测表现 |
| 长期胜率 | 长期预测表现 |
| 虚拟交易 ROI | 按其观点交易的模拟收益 |

加权胜率：

$$
WeightedWinRate = \frac{\sum Score_i}{N}
$$

如果分析师：

- 预测成功：提高胜率
- 预测失败：降低胜率
- 验证前改变观点：降低稳定性评分
- 多次模糊表达：降低可验证性评分

---

## 7. 观点改变处理模块

这是你需求里的重点。

系统需要检测：

1. 同一个分析师
2. 同一个时间周期
3. 同一个标的 BTC
4. 在原预测验证前
5. 是否出现方向反转或目标价大幅改变

例如：

原预测：

> 短期看涨到 80000

新预测：

> 短期可能跌到 75000

系统判断：

- 方向从看涨变成看跌
- 原预测状态改为“已修改/被放弃”
- 分析师稳定性扣分
- 新预测作为独立预测重新开始验证
- 虚拟交易中原多单应平仓，或反手开空

# 五、虚拟 BTC 合约交易模块

每个分析师建立一个虚拟账户。

## 1. 虚拟账户

每个分析师拥有：

- 初始资金，例如 10000 USDT
- 当前权益
- 可用保证金
- 持仓方向
- 杠杆倍数
- 已实现盈亏
- 未实现盈亏
- 手续费
- 资金费率成本
- 最大回撤
- ROI

---

## 2. 开仓规则

根据分析师预测自动模拟交易。

| 预测 | 虚拟交易行为 |
|---|---|
| 看涨 | 开多 BTCUSDT 永续合约 |
| 看跌 | 开空 BTCUSDT 永续合约 |
| 震荡 | 不交易或做区间策略 |
| 观点反转 | 平仓，必要时反手 |
| 到达目标价 | 止盈 |
| 到达失效条件 | 止损 |
| 到验证时间未达成 | 平仓 |

---

## 3. 手续费和资金费率

手续费参考币安 USDT-M 合约：

- 默认按 taker 手续费计算
- 可以在系统设置里配置 maker/taker 费率
- 如果未来你有币安账户等级，可以手动设置对应 VIP 费率

资金费率：

- 使用币安历史资金费率
- 通常每 8 小时结算一次
- 多头/空头根据资金费率正负支付或收取

---

## 4. 虚拟交易评价指标

每个分析师除了预测胜率，还要有交易表现：

| 指标 | 说明 |
|---|---|
| ROI | 总收益率 |
| 最大回撤 | 最大资金回撤 |
| 胜率 | 盈利交易 / 总交易 |
| 盈亏比 | 平均盈利 / 平均亏损 |
| 手续费总额 | 模拟手续费 |
| 资金费率成本 | 永续合约持仓成本 |
| 爆仓次数 | 风险指标 |
| 夏普比率 | 收益稳定性 |

这样可以区分：

- 预测经常对，但交易不赚钱的分析师
- 胜率一般，但盈亏比很好的分析师
- 经常改观点、不适合跟踪的分析师

# 十、推荐开发阶段

## 第一阶段：行情和数据库

目标：

- 从 Binance 拉取 BTC 历史价格
- 存入本地数据库
- 计算基础技术指标
- 前端展示 BTC K 线和指标

---

## 第二阶段：分析师观点录入

目标：

- 用户可以输入分析师观点
- LLM 自动解析
- 用户确认
- 生成结构化预测

---

## 第三阶段：预测验证和胜率

目标：

- 到时间自动验证预测
- 判断方向和目标价
- 更新分析师胜率
- 显示排行榜

---

## 第四阶段：观点变化检测

目标：

- 检测分析师是否改观点
- 对改观点行为扣分
- 维护预测版本历史

## 第五阶段：虚拟合约交易

目标：

- 每个分析师一个虚拟账户
- 根据预测开多/开空
- 计算手续费、资金费率、盈亏
- 展示 ROI 和回撤

---

## 第六阶段：BTC 综合分析 Agent

目标：

- 自动收集行情、技术指标、资金费率、新闻
- 生成 BTC 趋势分析报告
- 给出多空情景和关键价格

# 十一、最终系统效果

完成后，你的系统可以做到：

1. 自动存储 BTC 历史行情和技术指标  
2. 自动整理分析师观点  
3. 给每个分析师建立预测档案  
4. 自动验证预测是否正确  
5. 自动计算胜率和目标价准确率  
6. 检测分析师是否频繁改观点  
7. 为每个分析师建立虚拟合约交易账户  
8. 统计哪个分析师更值得参考  
9. 生成 BTC 趋势分析报告  
10. 前端可视化展示所有结果  

这个系统本质上不是单纯的聊天 Agent，而是一个“数据 + Agent + 评估 + 模拟交易”的 BTC 分析师追踪平台。
