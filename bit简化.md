# 一、项目定位

这个项目本质上是一个：

> **BTC 价格监控 + 分析师观点追踪 + 预测验证 + Agent 驱动模拟交易** 的轻量化智能系统

它不是完整量化交易平台，也不是重型数据中台，而是一个：

- 以 **LangGraph Agent** 为核心
- 以 **本地数据库** 为事实来源
- 以 **分析师观点** 为决策输入
- 以 **虚拟交易结果** 反向评估 Agent 决策质量

---

# 二、简化后的整体架构

推荐结构：

前端 Web  
→ 后端 API  
→ LangGraph Agent  
→ 本地数据库  
→ 币安行情数据源

可以理解为 4 层：

## 1. 前端层
负责展示和录入，不做复杂逻辑。

## 2. API 层
负责接收请求、返回数据、调度任务。

## 3. Agent 层
负责：
- 解析分析师观点
- 汇总市场信息
- 生成分析结论
- 决定虚拟交易下单
- 跟踪预测是否失效

## 4. 数据层
负责：
- 存 BTC 历史行情
- 存技术指标
- 存分析师观点
- 存预测记录
- 存验证结果
- 存虚拟交易记录

# 三、设计原则

## 1. Agent 为核心
系统不是“数据库驱动”，而是“Agent 决策驱动”。

Agent 是整个系统的大脑，数据库是它的记忆。

---

## 2. 后端中度轻量化
只保留 5 类核心能力：

- 行情采集
- 观点解析
- 预测记录
- 结果验证
- 交易模拟

不做过多复杂微服务。

---

## 3. 前端中度轻量化
前端只需要做到：

- 看行情
- 录入分析师观点
- 看分析结果
- 看预测验证
- 看 Agent 虚拟交易表现

不需要复杂下单系统、风控系统、深度交易面板。

---

## 4. 虚拟交易并入 Agent
这是你新的要求，应该这样理解：

> 虚拟交易不是一个独立业务模块，而是 Agent 决策链中的一个行为结果。

也就是说：

- Agent 看完行情和分析师观点后
- 自主决定是否开多、开空、平仓、反手
- 系统只负责记录它的行为和盈亏

# 四、简化后的核心模块

---

## 模块 1：行情数据模块

这是基础模块，只做一件事：

> 从币安持续获取 BTC 数据，存入本地数据库。

建议只保留必要数据：

- BTCUSDT 现货 K 线
- BTCUSDT 永续合约 K 线
- 成交量
- 资金费率
- 标记价格

技术指标不用存太多，先保留基础几个即可：

- MA / EMA
- RSI
- MACD
- ATR
- Bollinger Bands

这些指标是给 Agent 用来判断趋势的。

---

## 模块 2：分析师观点解析模块

用户输入分析师观点后，系统要做：

1. 识别分析师是谁
2. 提取方向
3. 提取目标价格
4. 提取时间周期
5. 提取观点是否明确
6. 生成结构化记录

例如原文：

> 某分析师认为 BTC 短期可能涨到 80000，中期可能回落到 75000。

系统应拆成两个结构：

- 短期看涨，目标 80000
- 中期看跌或回调，目标 75000

这部分建议由 Agent + 规则共同完成：

- 规则负责基本字段识别
- LLM 负责语义理解和总结

---

## 模块 3：预测记录与验证模块

每条分析师观点，会生成一条或多条预测记录。

每条预测记录至少包含：

- 分析师
- 发布时间
- 当前价格
- 方向
- 目标价
- 周期
- 验证时间
- 状态
- 是否被修改
- 是否成功

然后系统会在指定时间自动验证：

- 方向是否对
- 目标价格是否接近或触达
- 预测有没有被中途改口

---

## 模块 4：分析师评分模块

这个模块不做得太复杂，建议先做 4 个核心指标：

- 方向胜率
- 目标价命中率
- 观点稳定性
- 虚拟交易收益率

这样就足够了。

你原本提出的“胜率增减机制”保留，但简化成：

- 预测成功 → 加分
- 预测失败 → 扣分
- 验证前改观点 → 额外扣分
- 交易结果盈利 → 加分
- 交易结果亏损 → 扣分

建议最终生成一个综合评分，而不是只看单一胜率。

---

## 模块 5：Agent 决策与虚拟交易模块

这是整个项目的核心。

### 作用
Agent 在每个周期会读取：

- BTC 行情
- 技术指标
- 分析师观点
- 历史预测表现
- 当前未完成预测
- 市场波动情况

然后做出决策：

- 开多
- 开空
- 平仓
- 反手
- 观望

---

### 交易逻辑建议
Agent 的下单逻辑不要复杂化，先做轻量版：

#### 情况 1：看涨信号强
Agent 开多

#### 情况 2：看跌信号强
Agent 开空

#### 情况 3：观点冲突大
Agent 观望

#### 情况 4：已有持仓但观点反转
Agent 平仓或反手

---

### 交易记录
虽然虚拟交易并入 Agent，但仍需保存：

- 开仓时间
- 开仓价格
- 方向
- 仓位大小
- 平仓时间
- 平仓价格
- 手续费
- 资金费率
- 盈亏

这些数据用于评估 Agent 的交易能力。

---

# 五、LangGraph Agent 的轻量化设计

建议不要设计太多 Agent，做成一个主 Agent + 几个节点就够了。

---

## Agent 主体
主 Agent 负责协调整个流程。

它不是一个纯聊天机器人，而是一个“决策编排器”。

---

## 推荐的 LangGraph 节点

### 节点 1：行情摘要节点
输入：
- 最新 BTC 数据
- 技术指标

输出：
- 当前趋势判断
- 波动状态
- 关键支撑阻力

---

### 节点 2：分析师观点总结节点
输入：
- 分析师原文
- 历史观点

输出：
- 标准化总结
- 预测方向
- 目标价
- 周期
- 是否冲突

---

### 节点 3：预测一致性判断节点
输入：
- 新观点
- 旧观点
- 当前未验证预测

输出：
- 是否改口
- 是否需要扣分
- 是否生成新预测

---

### 节点 4：交易决策节点
输入：
- 行情摘要
- 观点总结
- 预测一致性结果
- 历史胜率
- 当前持仓

输出：
- 做多 / 做空 / 平仓 / 观望

---

### 节点 5：结果记录节点
输入：
- Agent 决策
- 价格后续变化
- 交易结果
- 预测验证结果

输出：
- 数据入库
- 分析师评分更新
- Agent 表现更新

# 六、Agent 决策逻辑建议

为了符合 Harness Engineering 思想，但又保持轻量，建议把 Agent 的行为约束成固定流程。

## 1. 输入受控
Agent 只能读：

- 行情
- 指标
- 分析师观点
- 预测历史
- 当前仓位

不能自由联网乱搜。

---

## 2. 输出受控
Agent 必须输出结构化结果：

- 市场判断
- 交易动作
- 理由摘要
- 风险说明
- 是否执行

---

## 3. 行为可追踪
每次 Agent 决策都保存：

- 输入
- 输出
- 触发原因
- 实际结果

这样便于后续评估。

---

# 七、后端设计方案

后端建议保持单体式，别拆太重。

推荐：

- FastAPI 作为 API 服务
- LangGraph 作为 Agent 工作流
- SQLite 或 PostgreSQL 作为数据库
- 定时任务作为数据同步和验证执行器

---

## 后端保留的核心能力

### 1. 行情同步
定时拉取币安 BTC 数据并入库。

### 2. 观点接收
接收前端输入的分析师文本。

### 3. 观点解析
调用 Agent 或 LLM 进行结构化总结。

### 4. 预测生成
把总结结果写入预测表。

### 5. 定时验证
到期后自动检查预测结果。

### 6. Agent 决策
读取最新数据，决定虚拟交易行为。

### 7. 结果评估
更新分析师表现和 Agent 表现。

# 十、虚拟交易如何并入 Agent

这是重点，我单独说明。

## 传统方式
通常是：
预测模块 → 交易模块 → 账户模块

## 你这个项目的方式
应该改成：

预测和行情 → Agent 判断 → Agent 直接给出交易动作 → 系统记录交易结果

也就是说：

- 交易不是单独的“模块”
- 交易是 Agent 的一个“动作结果”

---

## 这样设计的好处

1. 更符合 agent 项目定位
2. 架构更轻
3. 更适合验证“Agent 是否能靠分析信息做出合理行为”
4. 便于后面评估 Agent 本身，而不是只评估交易模块
